1.Overview & Goals

The PR Review Agent is a hosted web service that performs automated code review of public GitHub pull requests. A client submits a PR URL over HTTP; the service fetches the diff and surrounding file context via the GitHub REST API, sends it through a Gemini-powered review pipeline, and returns a structured review: an overall verdict, a summary, and per-file findings with severity, category, and suggested fixes.

Goals

  • Accept any public GitHub PR URL and return a useful, structured review.
  • Review quality beyond diff-only: include the full file content at head SHA for changed files so the model sees surrounding context.
  • Simple, stable HTTP API that other tools (dashboards, bots, scripts) can integrate with.
  • Handle large PRs gracefully via chunked per-file review passes and explicit token budgets.
  • Operate within free/low-tier GitHub and Gemini rate limits; degrade predictably when limits are hit.

Non-Goals (v1)

  • Private repositories or GitHub OAuth flows.
  • GitLab / Bitbucket support.
  • Posting review comments back to the PR on GitHub.
  • CI/webhook-triggered automatic reviews.
  • Cloning repositories or running the code/tests.

All of these are candidates for Future Work (§9).

2.High-Level Architecture

The service is a single deployable (FastAPI app) with five logical components. Reviews are asynchronous jobs: a submission immediately returns a job ID, and the client polls for the result. This avoids long-held HTTP connections, since reviewing a mid-sized PR takes 20–90 seconds.

Processing model

  • POST /api/v1/reviews validates the URL, creates a job record, and enqueues the review as a FastAPI background task (v1) — no external queue needed.
  • The background task runs the pipeline: fetch → build context → review → format, updating job status at each stage.
  • Results are cached by PR head SHA: re-submitting the same PR at the same commit returns the finished review instantly.

    "findings": [
      {
        "file": "app/webhooks/payment.py",
        "line": 88,
        "severity": "blocker",
        "category": "bug",
        "comment": "Retry fires before the idempotency key is persisted, so a timeout can charge twice.",
        "suggested_fix": "Persist the idempotency key before calling the gateway; retry only on read of an unpersisted key."
      }

  }
}

All errors use a single envelope: { "error": { "code", "message", "details?" } }.

4.Component Design

4.1 PR Fetcher

Parses and validates the PR URL, then reads everything needed through the GitHub REST API (read-only, no clone):

EndpointProvides
GET /repos/{o}/{r}/pulls/{n}Title, description, base/head SHAs, change stats, state.
GET /repos/{o}/{r}/pulls/{n}/files (paginated)Per-file status (added/modified/removed/renamed) and unified patch hunks.
GET /repos/{o}/{r}/contents/{path}?ref={head_sha}Full file content at the PR head, for context.
Rate limits: unauthenticated GitHub API calls are capped at 60 requests/hour per IP — enough for roughly 2–4 reviews. The service accepts an optional GITHUB_TOKEN (any classic PAT with public-repo scope) raising this to 5,000/hour. This is strongly recommended for any real deployment and is the default assumption in capacity planning below.
  • Skip rules: binary files, generated files (lockfiles, *.min.js, dist/), and files over 4,000 lines are excluded from model input but listed as files_skipped in the result.
  • Retries: exponential backoff (3 attempts) on 5xx and secondary-rate-limit responses; hard fail on 403 quota exhaustion with rate_limited.

4.2 Context Builder

Assembles model input per file: the diff hunks plus the full file at head SHA, so the model sees the code around each change. Works to an explicit token budget:

  • Budget: ~150k input tokens per model call (well under Gemini's 1M window, keeping latency and cost predictable).
  • Batching: files are greedily packed into batches under the budget; small PRs review in a single call, large ones in several parallel calls.
  • Truncation: when a full file would blow the budget, it degrades to diff hunks ± 60 lines of surrounding context, and the finding notes reduced context.
  • PR metadata (title, description, file list) is included in every batch so each call understands the intent of the whole change.

4.3 Review Engine (Gemini)

Per-batch review pass — gemini-2.5-pro

One call per file batch. The prompt gives the model a senior-reviewer persona, the PR intent, the batched files (diff + context), review categories, and severity definitions. Output is constrained JSON via Gemini's response_schema — a list of findings matching the schema in §5. No free-text parsing.

Synthesis pass — gemini-2.5-flash

One cheap call that receives all findings + PR metadata and produces the overall verdict and 3–6 sentence summary, deduplicating findings that multiple batches reported. Flash is sufficient because this pass reasons over findings, not code.

  • Prompt rules baked in: comment only on changed code (context is for understanding, not for drive-by refactors); every finding must cite file + line from the diff; prefer concrete suggested fixes; do not invent line numbers.
  • Determinism: temperature=0.2; findings failing schema validation trigger one re-ask, then are dropped with a logged warning.
  • Categories: bug, security, performance, error_handling, maintainability, testing, style.

4.4 Response Formatter

  • Merges batch findings, sorts by severity then file path, and maps diff-relative positions to head-file line numbers.
  • Derives the verdict deterministically as a floor: any blocker ⇒ request_changes; only minor/info ⇒ approve_with_nits; the synthesis pass may downgrade but never upgrade this.
  • Writes the completed result (or failure) into the Job Store and stamps duration_ms and call counts.

5.Data Model

5.1 ReviewJob

ReviewJob {
  review_id:   string       // "rev_" + 8-char random id
  status:      enum         // queued | fetching | reviewing | completed | failed
  pr:          PRRef        // owner, repo, number, title, head_sha, stats
  focus:       string[]     // requested categories, empty = all
  result:      ReviewResult | null
  error:       ErrorInfo | null
  created_at / started_at / finished_at: timestamp
}

5.2 Finding

Finding {
  file:           string    // path at head SHA
  line:           int | null // head-file line; null for file-level findings
  severity:       enum      // blocker | major | minor | info
  category:       enum      // bug | security | performance | error_handling | maintainability | testing | style
  comment:        string
  suggested_fix:  string | null
}

7.Non-Functional Concerns

7.1 Rate limiting & quotas

  • Inbound: per-IP throttle (e.g. 10 submissions/hour) plus a global concurrency cap (default 3 concurrent reviews) to protect upstream quotas.
  • GitHub: with a token, 5,000 req/hr. A 30-file PR costs ~35 requests (1 PR + 2 pages of files + 30 contents), so the ceiling is ~140 mid-size reviews/hour — inbound throttling keeps us far below it.
  • Gemini: respect RPM/TPM limits of the key's tier; the concurrency cap plus batch sizing keeps a review to typically 2–5 model calls.

7.2 Caching

  • Completed reviews cached by (owner, repo, number, head_sha) for 24h — re-submitting an unchanged PR is free and instant.
  • A new push changes the head SHA and naturally invalidates the cache.

7.3 Reliability

  • Timeouts: 30s per GitHub call, 120s per Gemini call; the whole pipeline is capped at 10 minutes before the job fails.
  • Retries with exponential backoff on transient upstream errors; jobs fail closed with a structured error, never hang in reviewing.
  • v1 job store is in-memory: results are lost on restart (acceptable — reviews are cheap to re-run and cached by SHA). Redis is the drop-in upgrade when persistence matters.

7.4 Security & secrets

  • GEMINI_API_KEY and optional GITHUB_TOKEN come from environment variables (or the platform's secret manager); never logged, never echoed in responses.
  • Only github.com PR URLs are accepted (strict parse, no redirects followed) — prevents SSRF via attacker-supplied URLs.
  • Reviewed code is untrusted input to the model: prompts instruct the model to treat file content purely as data, and outputs are schema-validated, which bounds prompt-injection blast radius (worst case: bad review text, no tool access to escalate).
  • If the API is exposed publicly, put a simple API key check (X-API-Key) in front of POST /reviews.

9.Future Work

  1. GitHub App integration — post the review back as inline PR comments and a review summary; reuse the Formatter output, add a Commenter component.
  2. Webhook-triggered reviews — auto-review on pull_request events for installed repos.
  3. Private repositories — GitHub App installation tokens replace the public-only fetcher path.
  4. GitLab / Bitbucket — extract a Provider interface from the PR Fetcher (URL parse, metadata, files, contents).
  5. Deeper analysis — optional shallow clone for cross-file symbol tracing; run linters and include their output as model context.
  6. Incremental re-review — on a new push, review only files changed since the last reviewed SHA.
